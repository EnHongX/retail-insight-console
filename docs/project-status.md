# 项目开发进度

更新日期：2026-06-22

## 该不该做

应该继续推进，但下一阶段不应扩展成通用 BI。

当前 MVP 已经具备可演示的运营闭环：看核心指标、筛选数据、查看订单明细、处理退款。下一步的价值在于把它从“本地样例可跑”推进到“准上线可用”，重点是数据导入、运行环境、异常处理和上线前安全边界。

## 为什么

- 已完成的功能足够支撑一次面向运营的端到端演示。
- 最大缺口不在更多图表，而在真实数据进入、部署配置、错误可观测和基础访问控制。
- 继续堆复杂报表会进入 ⚠️ 技术过度设计 / 收益不匹配；先补上线能力更划算。

## 当前进度

### 已完成

- Monorepo：pnpm workspace 管理 `apps/api` 和 `apps/web`。
- 后端：NestJS API，健康检查，Dashboard、订单、退款、商品排行接口。
- 数据层：PostgreSQL + Prisma schema、初始化 migration、seed 数据。
- 指标：GMV、订单数、退款金额、退款率、客单价、销售趋势、商品销售排行。
- 筛选：周期、平台、分类、商品搜索、退款状态、商品排行排序。
- 订单明细：支持分页、趋势日期 Drilldown、展开订单商品和关联退款。
- 退款工作台：支持 `REQUESTED`、`APPROVED`、`REJECTED`、`COMPLETED` 状态流转。
- 前端：React + Vite 看板，三视图导航：总览、订单明细、退款处理。
- 测试：纯函数指标、筛选序列化、视图导航、退款状态机已有单测。

### 已验证

2026-06-22 本地验证结果：

- `pnpm typecheck`：通过。
- `pnpm build`：通过。
- `pnpm test`：通过，5 个测试文件，18 个测试。
- `pnpm validate:domain`：通过。
- `pnpm db:validate`：通过。

### 已知风险

- ⚠️ 时间黑洞：如果下一步直接做自定义报表、拖拽 BI、多租户，会明显偏离 MVP 上线目标。
- ⚠️ 伪需求：当前还没有真实运营用户反馈，不宜继续增加图表种类。
- Prisma 提示 `package.json#prisma` 配置将在 Prisma 7 废弃，升级前需要迁移到 Prisma config 文件。
- 当前没有认证、权限、审计日志，不适合直接暴露到公网。
- 当前数据来自 seed，缺少真实订单 CSV/API 导入链路。
- 当前缺少端到端浏览器测试和 API 集成测试，改动 UI/API 合同时回归成本较高。

## 下一步开发计划

### P0：准上线基础

1. 增加真实数据导入入口，优先 CSV 导入订单、商品、退款，不先接平台 OpenAPI。
2. 增加基础登录保护，最小可行方案为单管理员账号或部署层 Basic Auth。
3. 增加生产环境配置文档：数据库、环境变量、迁移、seed、启动命令。
4. 增加 API 错误响应规范，避免前端只显示通用失败文案。

### P1：质量和回归

1. 补 API 集成测试，覆盖 Dashboard 汇总、订单分页、退款状态流转。
2. 补前端端到端冒烟测试，覆盖筛选、趋势点击、退款状态更新。
3. 固化测试数据库启动方式，避免依赖开发者本机已有 PostgreSQL。
4. 处理 Prisma 7 配置迁移提示。

### P2：产品增强

1. 增加运营常用导出：订单明细 CSV、退款明细 CSV。
2. 增加异常提示：退款率突增、GMV 环比下降、待处理退款积压。
3. 增加商品维度下钻，但保持固定分析路径，不做自由报表搭建器。

## 暂不做

- 多租户。
- 复杂权限体系。
- 实时数据同步。
- 自定义 BI / 拖拽报表。
- 库存、会员、营销活动全域模型。
