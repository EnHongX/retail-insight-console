# 电商运营 MVP 指标口径

## 目标

第一版只回答一个问题：运营今天和最近一段时间的销售、退款、商品表现是否正常。

不做复杂 BI、不做自定义报表、不接真实平台 API。数据先来自 seed 或后续 CSV 导入。

## 时间口径

- 默认按订单支付/下单时间 `orders.placedAt` 统计销售类指标。
- 退款类指标按退款完成时间 `refunds.completedAt` 统计；未完成退款不计入已退款金额。
- Dashboard 第一版支持今日、近 7 天、近 30 天。
- 日期边界按服务端时区处理，后续上线前再统一为店铺时区。

## 核心指标

### GMV

GMV = 已支付、已发货、已完成订单的商品成交金额总和。

使用字段：`orders.grossAmount`。

不计入：

- 已取消订单
- 待支付订单
- 售后退款抵扣

### 订单数

订单数 = 已支付、已发货、已完成订单数量。

使用字段：`orders.status`。

### 退款率

退款率 = 已完成退款金额 / GMV。

使用字段：

- 分子：`refunds.amount`，仅 `refunds.status = COMPLETED`
- 分母：同期 GMV

### 客单价

客单价 = 净销售额 / 订单数。

使用字段：`orders.netAmount`。

净销售额暂不扣除已完成退款。退款影响单独通过退款率呈现，避免第一版口径混杂。

### 商品销售排行

商品销售排行默认按销量排序。

使用字段：

- 销量：`order_items.quantity`
- 销售额：`order_items.netAmount`

仅统计有效订单：已支付、已发货、已完成。

## 最小数据模型

- 商品：SKU、名称、分类、品牌、标价、成本、状态。
- 订单：订单号、平台、下单时间、状态、成交金额、优惠金额、净销售额。
- 订单明细：订单和商品的多对多明细，记录数量、单价、优惠、行净额。
- 退款：退款号、关联订单/明细、退款原因、状态、申请时间、完成时间、金额。

## 暂不做

- 多店铺、多租户。
- 用户权限。
- 实时数据同步。
- 可视化报表搭建器。
- 库存流水。

## 评测夹具

Seed 数据需要稳定覆盖以下场景：

- 近 60 天每天至少有一笔订单，且今天和近 7 天有有效订单，销售趋势可以看出波动。
- 同一天存在多个平台订单，平台筛选会改变 GMV 和商品排行。
- 同一商品跨多日、多平台出现，商品搜索和排行排序能产生差异。
- 存在 `CANCELLED`、`PENDING_PAYMENT` 订单，验证有效订单过滤。
- 存在完成退款、待审核退款、已批准退款、已拒绝退款。
- 至少一笔退款的 `requestedAt` 和 `completedAt` 不在同一天，验证退款完成时间口径。

## 交互链路

### 订单明细 Drilldown

- 点击销售趋势中的日期后，订单明细切换为该日订单。
- 明细继承平台、分类、商品搜索等筛选条件。
- 订单可展开查看商品明细和关联退款。
- 分页切换不得丢失当前筛选和选中日期。

### 退款处理工作台

- 工作台继承 Dashboard 当前筛选和退款状态筛选。
- 可执行 `Approve`、`Reject`、`Complete`。
- 合法状态流转：
  - `REQUESTED -> APPROVED`
  - `REQUESTED -> REJECTED`
  - `APPROVED -> COMPLETED`
  - `APPROVED -> REJECTED`
- `REJECTED` 和 `COMPLETED` 是终态。
